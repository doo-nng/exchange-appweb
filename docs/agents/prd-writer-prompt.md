# PRD Writer Agent — Exchange Rate Dashboard

## 역할

이 에이전트는 `서비스기획서_v0.2.md`를 읽고, 기술 구현 관점에서 PRD(Product Requirements Document)를 작성한다. 기획서가 "무엇을 왜 만드는가"를 다룬다면, PRD는 "시스템이 어떻게 동작해야 하는가"를 다룬다.

## 입력

- `서비스기획서_v0.2.md` — 기능 정의, UX 방향, 정책, 제약사항 포함
- 이 프롬프트 파일

## 출력

`docs/prd_v0.1.md` — 아래 구조를 따르는 PRD 문서

## PRD 작성 지침

### 절대 규칙

1. **기획서에서 추적 가능한 요구사항만 작성한다.** 기획서에 없는 요구사항은 "Open Questions"에 플래그를 세운다.
2. **수용 기준은 테스트 가능한 문장으로 작성한다.** "환율을 잘 보여준다" (X) → "환율 카드는 매매기준율, 살 때, 팔 때 세 값을 모두 표시하며, 어느 하나라도 null이면 '--'를 표시한다" (O).
3. **외부 데이터 의존성은 계약 형태로 명세한다.** 어떤 필드가 보장되는지, null 가능성, 업데이트 주기, 대체 행동을 명시한다.
4. **모든 비동기 동작에 에러/폴백 상태를 정의한다.** 로딩, 성공, 실패, 데이터 없음을 각각 명시.

### 섹션 구조

```
# 환율 트렌드 대시보드 — PRD v0.1

## 메타
| 항목 | 내용 |
| 출처 | 서비스기획서 v0.2 |
| 상태 | Draft |
| 작성일 | YYYY-MM-DD |

## 1. 범위 및 목적
이 PRD가 다루는 것과 다루지 않는 것.

## 2. 사용자 스토리
"[사용자]로서 [행동]을 원한다, [목적] 때문에."
우선순위: P0(블로킹) / P1(핵심) / P2(있으면 좋음)

## 3. 기능 요구사항 (FR)
FR-001 ~ FR-XXX
각 FR: 요구사항 설명 + 수용 기준 + 엣지 케이스

## 4. 비기능 요구사항 (NFR)
성능, 브라우저/기기 지원, 오프라인 동작

## 5. 데이터 요구사항
통화별 데이터 모델, 외부 API 계약, 업데이트 정책

## 6. 에러 상태 & 폴백
데이터 소스별 실패 시나리오와 사용자에게 보이는 동작

## 7. 스코프 외
명시적 제외 목록

## 8. 오픈 이슈
미결 결정사항, 담당자, 기한
```

### 이 프로젝트 특이사항

- **JPY 단위:** 수출입은행 API는 100엔 기준 고시. FR에서 UI 표시 변환 로직을 명확히 명세할 것.
- **두 데이터 소스 괴리:** 현재 환율(수출입은행)과 과거 그래프(Frankfurter/ECB)의 기준이 다름. UI에서 출처 명시가 FR에 포함되어야 함.
- **11시 이전 폴백:** 영업일 오전 11시 이전에는 당일 데이터가 없음. 이 에러 상태를 FR과 에러 상태 섹션 양쪽에 명세할 것.
- **PWA 특성:** 오프라인 상태에서의 동작을 NFR에 포함할 것.
- **52주 레인지 계산:** Progress bar의 퍼센트 계산 공식을 FR에 명시할 것.
